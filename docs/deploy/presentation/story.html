01 The pipeline, as a story

Merge to main.
It's live in under 5 minutes.
Zero hands.

A pull-based, hands-off continuous deployment pipeline for RobotOverview — Next.js 16.2 on a single-node k3s cluster called bee-mini. No manual docker push. No kubectl apply. No second PR.

Scroll right (or press Play) to see how a mutable tag broke everything — and the one-line insight that fixed it.

02 The problem

A mutable tag never redeploys.

The original setup pinned a mutable tag :0.1.0 with imagePullPolicy: IfNotPresent. New code merged to main… and never reached the pod.

ARGO SEES NO CHANGE

Argo CD reconciles git text. The tag :0.1.0 never changes, so the manifest never changes — so Argo never triggers a rollout.

NODE CACHES FOREVER

IfNotPresent means the node keeps the first image it pulled. Even if it did roll, it would never re-pull the same tag.

  • New code merged, but the live pod kept serving the old bits.
  • Two independent reasons stacked: unchanged git text + cached image.
03 The insight

Argo CD watches git, not the registry.

That's the whole key. Pushing a new image to GHCR is invisible to Argo. The only thing Argo reacts to is a change in the desired-state manifest sitting in the GitOps repo.

So "auto-deploy" reduces to one job:
automatically change the image reference in git.

Once a release writes a new reference into git, Argo's normal reconcile loop does the rest. No new in-cluster machinery required.

04 The fix

CI writes the digest into the GitOps repo.

The CI-writes-to-GitOps pattern — Argo CD's own documented CI flow. On merge to main, CI builds & pushes the image, then commits the new @sha256 digest directly into the GitOps repo's main.

  • Two repos. Coldaine/RobotOverview (app + CI) writes into Coldaine/coldaine-k8cluster (the state Argo watches).
  • Bot commit, no PR. Committed as robot-overview-delivery-bot with [skip ci].
  • Immutable digest pin. A unique @sha256 per release changes the manifest every time → guaranteed rollout, exact bits, rollback = git revert.
  • Pull-only cluster. CI never reaches into bee-mini; the node only pulls git + images. No inbound path added.
05 The flow

Build → push → pin digest → commit → Argo → pod.

CI only commits; it never runs an explicit argocd app sync — combining that with auto-sync is the documented race-condition footgun, so we avoid it.

06 Why not the alternatives

Two obvious tools — both deliberately rejected.

argocd-image-updater
  • Decisive: only supports Kustomize/Helm — not plain-YAML / Directory apps, which robot-overview is. (Still true June 2026.)
  • Needs a separate write-capable git credential; Argo's deploy key is read-only.
  • Rejected on the plain-YAML limit — not on "not prod-ready." Do not install it.
Flux image-automation
  • Would mean running a second GitOps engine next to Argo — on a single node.
  • Overkill for the job.

CI-writes-to-GitOps is the canonical Argo CD CI flow and the lowest surface area: zero in-cluster components, works the same for plain-YAML & Kustomize, and fails loudly in an Actions log.

08 How to monitor it

Four checks, top to bottom.

  • Build + write-back ran? RobotOverview → Actions → the update-k8cluster job log shows the pinned digest (or the no-op notice).
  • Digest landed in git? coldaine-k8cluster main history — a deploy <sha> [skip ci] commit by robot-overview-delivery-bot.
# Did Argo sync?
kubectl get application robot-overview -n argocd
  # expect: Synced / Healthy

# New pod on the new digest?
kubectl -n robot-overview get pods
  # expect: 1/1 Running

kubectl -n robot-overview get deploy \
  robot-overview -o jsonpath='{..image}'
  # → ...@sha256:<digest>

Serving check — publicly via the Cloudflare Tunnel hostname; in-cluster at robot-overview.robot-overview.svc.cluster.local:80. (argocd CLI isn't installed — use kubectl.)

09 Setup status

Three steps arm the GitHub App.

Until both secrets exist, the update-k8cluster job no-ops gracefully (checks APP_READY = both present) — by design, so the workflow never hard-fails before setup.

1
Create a GitHub App. Repository permission Contents: Read & write only — installed on only Coldaine/coldaine-k8cluster.
2
Add two repo secrets to RobotOverview. GITOPS_APP_CLIENT_ID (the App's Client ID — create-github-app-token@v3 uses client-id) and GITOPS_APP_KEY (the PEM private key). Remove the old K8CLUSTER_REPO_TOKEN PAT.
3
Allow the bot to push to coldaine-k8cluster main. Add the App to the branch-protection bypass/allowlist (direct push, no PR).
Once armed: merge → live in < 5 min, hands-off.
